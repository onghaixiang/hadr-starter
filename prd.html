01 — Executive Summary

What we are building, and why now

Every morning at 08:30 Singapore time, an agent watches the world's live disaster feeds, works out what happened overnight in the ASEAN neighbourhood — what, where, how bad, who is affected — and publishes it to dashboard.html. When nothing significant has changed, it still publishes an explicit “all quiet, feeds healthy” state. Silence is a reported signal, not an absence of one.

Humanitarian assistance and disaster response (HADR) teams — militaries, UN agencies, NGOs — share a common first problem when disaster strikes: knowing that something has happened, how bad it is, and who is affected, fast enough to act. The hours after an event are spent answering exactly those questions from noisy, conflicting sources. This agent takes on that job for the start of the day: it separates signal from noise across three feeds and has an assessment ready before the desk sits down.

The value is trust through speed. A watch officer should not have to open three portals, reconcile three severity vocabularies, and mentally diff against yesterday. The report does that work deterministically overnight and presents a triage-ready briefing — including its own health, so a degraded feed is visible rather than silently missing.

02 — Problem Statement

The situation, the solution, the impact

Current situation

A duty officer starting the day has no single, trustworthy overnight briefing scoped to the region. The raw sources are fast but noisy (USGS emits ~200 events/day, mostly Californian microquakes), modeled but undocumented (GDACS), or authoritative but slow (ReliefWeb, days behind). Their severity scales are not translations of one another, they revise events after publication, and every endpoint is a rolling window that cannot be replayed. Reconciling this by hand, daily, before 09:00, is exactly the fragile manual work that gets skipped under load.

Proposed solution

A scheduled GitHub Actions run in which (1) a deterministic script decides whether anything changed, and (2) a headless model call runs only if it did, to write the human-facing prose. The model never decides whether to wake up. Everything that must give the same answer twice — change detection, dedup, severity thresholds — lives in scripts/, not a prompt. The output is a committed dashboard.html scoped to ASEAN, at an impact-modeled noise floor.

Business impact

One reliable, region-scoped briefing replaces an ad-hoc multi-portal scan, cutting time-to-first-assessment and — more importantly — removing the risk that a significant overnight event is missed because no one checked. An explicit health state converts silent feed outages into reported gaps, protecting the credibility the product lives or dies by.

03 — Success Metrics

How we will know it works

The primary KPI is triage speed: the report earns its place only if it lets a duty officer decide fast. Completeness and reliability are the guardrails that make that speed trustworthy.

Primary KPI

Fast triage < 2 min

A duty officer can decide “needs attention today or not” within ~2 minutes of opening the report. Measured by structured read-throughs on days with 0, 1, and several events; validated by the desk confirming they didn't need to open another source to make the call.

Guardrail · Trust

No missed significant events 0 false-neg

Every in-region event above the noise floor (GDACS Orange+, PAGER yellow+, or populated-area quakes) appears in the next report. Measured by back-testing the deterministic filter against recorded fixtures for known events; a miss is a defect, not a tuning note.

Guardrail · Reliability

Publishes every morning ≥ 99%

The 08:30 SGT report lands unattended every day, including a correct “all quiet, feeds healthy” state and a visible signal when any feed is stale or down. Measured by scheduled-run success rate and presence of a per-source freshness line in every published report.

Validation cadence: filter back-tests run in CI on every change (against fixtures, never live feeds); triage-speed and reliability are reviewed at the Day-3 “Trust” demo and thereafter on any week the report is relied on.

04 — User Persona

Who reads this at 08:30

Primary — The Duty Officer / Watch Desk

  • Role: Starts a shift responsible for regional situational awareness (a HADR ops cell, national watch desk, or NGO regional coordinator).
  • Goal: Answer one question quickly — “What happened overnight that I need to know about?” — and decide whether any event warrants attention today.
  • Pain points: Multiple portals with incompatible severity scales; noise drowning signal; no memory of “what changed since yesterday”; no way to tell a quiet night from a dead feed.
  • Technical level: Intermediate. Comfortable reading structured briefings and technical provenance; does not want to operate tooling or run queries at 08:30.
  • Lens: Global in style, but focused on the ASEAN neighbourhood (Singapore-centric).
05 — User Stories & Acceptance Criteria

Core stories

Story 1 · Overnight briefing

Read what changed overnight

As a duty officer I want a single morning report of significant regional disaster activity so that I can decide what needs attention today without opening multiple sources.

  • Report is published to dashboard.html by 08:30 SGT, scoped to SEA/ASEAN.
  • Each event states what / where / how bad / who is affected, with a severity indication.
  • “New since the last report” is computed on first-seen / last-changed time, not event time — so a late-arriving event is never silently omitted.
  • The report is readable end-to-end in under ~2 minutes on a normal day.
Story 2 · Explicit quiet

Trust a quiet morning

As a duty officer I want the report to say explicitly when nothing significant changed so that I can distinguish a genuinely calm night from a broken pipeline.

  • On a no-change morning the report still publishes an explicit “all quiet, feeds healthy” state.
  • Every published report names each source's last-success time / freshness.
  • A stale or unavailable feed is reported as a gap, never silently dropped.
Story 3 · Signal over noise

See only what matters

As a duty officer I want the ~200/day raw tremors filtered to impact-relevant events so that the briefing is signal, not a firehose.

  • Filtering is impact-modeled (GDACS colour, PAGER/sig), not magnitude-alone.
  • Events below the defined noise floor are excluded from the narrative.
  • The filter is a deterministic script, covered by fixture-based tests.
Story 4 · Revisions as changes

Follow an event as it evolves

As a duty officer I want upgrades, downgrades and revised figures reported as changes so that yesterday's assessment is corrected rather than frozen.

  • Each run re-evaluates events first seen in a recent lookback window, not just new arrivals.
  • Threshold crossings in either direction, and deletions, are reported as changes (e.g. “deaths revised 12 → 89”).
  • Every volatile figure carries a source and an as-of time.
06 — Domain Model

Three feeds, one funnel

The three sources are not three views of the same thing. They are three stages of a funnel, each answering a different question at a different latency. Their severity vocabularies are not translations of each other, and a mismatch between them is often the funnel working correctly, not an error. Cross-feed “dedup” is really linking one physical event as it moves down the funnel.

Severity is reported on one internal scale with an explicit precedence rule for conflicts (GDACS colour wins where present; PAGER maps green → yellow/orange → red; magnitude alone never sets severity). Forecast alerts (GDACS cyclones) are reported as “expected”, never as having occurred.

07 — Functional Requirements

Core features

F1 · Feed ingestion & normalisation

Fetch each source behind a swappable adapter (GDACS JSON with RSS fallback). Normalise every payload into one internal event schema before it reaches diffing, rendering, or the prompt — downstream code never sees a raw feed format. Snapshot every raw payload on fetch (fixtures + audit trail). Edge cases: empty response is valid and must not read as an error; three timestamp dialects (epoch-ms UTC, naive-UTC strings, RFC-822) are all resolved against an explicit SGT reporting-day boundary.

F2 · Deterministic change detection

A script decides whether anything meaningful changed since the last report and exits in a way the workflow can branch on — without calling a model. “Changed” is computed on first-seen / last-changed, over a recent lookback window, so late arrivals and post-publication revisions are caught. Error handling: phantom “deletions” from a rolling window are distinguished from true deletions.

F3 · Noise filtering & severity

Apply the impact-modeled noise floor per feed (GDACS colour, PAGER/sig) and resolve severity onto one internal scale with the documented precedence rule. Recently-seen events are re-evaluated each run, so an upgrade of a previously-discarded event is not structurally missed.

F4 · Cross-feed identity & sequence grouping

Store all source IDs as an alias set. Match earthquakes across feeds by origin time / epicentre / magnitude tolerance; attach GLIDE as confirmation when it appears. Group aftershock sequences for reporting (“M7.5 + 47 aftershocks”) — distinct from identity matching. Bias: prefer an unmerged duplicate over a wrong merge.

F5 · Report rendering + all-quiet state

Render dashboard.html: triage-ordered events with what / where / how bad / who, provenance and as-of times on volatile numbers, and an explicit all-quiet state on no-change mornings. The model is invoked only when F2 signals a change, to write prose over already-decided facts.

F6 · Health surface

Track and render per-source last-success time and payload freshness. A stale/unavailable feed is surfaced as a reported gap in every report — the mechanism that makes silence trustworthy.

Out of scope (this release)

  • No alerting / paging / notification channel beyond the committed dashboard.html.
  • No historical replay UI or querying of past reports.
  • No user accounts, authentication, or per-user configuration.
  • No mutation handling beyond “latest snapshot wins” best-effort (some churn / occasional duplicate accepted).
08 — Technical Constraints

Constraints & integration

Architecture & determinism

  • Fixed shape: scheduled run → deterministic change-check script → headless claude -p only on change. The model never decides whether to wake up.
  • Determinism out of prompts: change detection, dedup, severity thresholds live in scripts/.
  • Fixtures-first: parsers are built and tested against committed fixtures; CI never calls live feeds.

Polling budget & performance

  • USGS ≤ 1/min with If-Modified-Since/ETag; GDACS ~10–15 min; ReliefWeb 3–4×/day.
  • Report generation is a once-daily batch — throughput is not a concern; correctness and reproducibility are.

Integration & endpoints

  • USGS: summary GeoJSON feed for polling; FDSN event query API for lookback / confirming true deletions.
  • GDACS: JSON event list behind an adapter, RSS/CAP as fallback (JSON endpoint is undocumented).
  • ReliefWeb: /reports + /disasters via API once appname lands; RSS with GLIDE-parsing until then. Hard cap 1000 calls/day.

Technology stack

  • Python, standard library first; dependencies pinned in requirements.txt only when they earn their place.
  • pytest from repo root; scheduled via GitHub Actions (mind the cron timezone vs 08:30 SGT).
  • No secrets beyond the ReliefWeb appname and the Claude key; dashboard.html is committed to the repo.

Security & compliance: all sources are public feeds; no PII is stored. The only credentials are API keys held as Actions secrets. Deviations from this PRD or CLAUDE.md are recorded in implementation-notes.md.

09 — Scope & Phasing

Vertical slices

The build cuts vertically — a thin end-to-end path first, then breadth. Slice 1 proves the whole architecture through a single feed before any cross-feed complexity is introduced.

MVP · Slice 1

USGS-only, end-to-end

Day 2 · first slice
  • Fetch USGS → normalise to the internal event schema → snapshot raw payload as fixture.
  • Deterministic change detection (first-seen / last-changed) + impact-modeled noise floor on sig/PAGER.
  • Render dashboard.html with the explicit all-quiet state and a per-source health line.
  • Wire the 08:30 SGT GitHub Actions routine: change-check gates the headless model call.

Definition of done for the MVP: on a quiet morning the routine publishes “all quiet, feeds healthy”; on an active morning it publishes a triage-ready USGS briefing — unattended, from committed fixtures in test.

Slice 2 — Add GDACS & cross-feed matching

Post-MVP
  • GDACS adapter (JSON + RSS fallback); colour-based severity onto the shared scale with precedence rule.
  • Cross-feed identity matching (alias sets, space-time tolerance) and aftershock-sequence grouping.
  • Re-evaluation of recently-seen events to catch post-publication colour upgrades.

Slice 3 — Add ReliefWeb & multi-hazard

Later
  • ReliefWeb /reports + /disasters once appname is approved; GLIDE as link confirmation.
  • Non-earthquake hazards (cyclone forecasts as “expected”, floods/droughts as spanning areas).
  • Richer provenance and revision tracking on volatile impact figures.

11 — Dependencies & Blockers

What we depend on

  • ReliefWeb appname — pre-approved, requested via form, confirmed by email (manual review). Owner: requested Day 1. Mitigation: RSS fallback; not on the MVP critical path.
  • Claude API key + GitHub Actions secrets — required for the headless report step and scheduled run.
  • Recorded feed fixtures — a committed snapshot per source is a prerequisite for any parser work (fixtures-first).
  • Live-feed availability & goodwill — no SLAs on GDACS/USGS; polite polling budgets are a hard constraint, not a preference.
